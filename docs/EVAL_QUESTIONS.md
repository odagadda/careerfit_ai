# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문



## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.



## 프로젝트 전체

* 이 서비스가 해결하는 문제는 무엇인가?

  → 취업 준비생이 수많은 채용 공고를 직접 비교하면서 자신에게 맞는 공고와 필요한 역량을 파악하기 어렵다는 문제를 해결합니다. CareerFit AI는 사용자의 전공, 보유 스킬, 관심 직무를 바탕으로 맞춤 공고와 부족한 역량을 알려줍니다.

* RAG를 사용하지 않으면 어떤 문제가 생기는가?

  → AI가 실제 채용 공고 데이터를 참고하지 않고 일반적인 답변만 생성할 수 있습니다. 그러면 추천 결과가 부정확하거나 근거가 부족해질 수 있고, 실제 공고와 맞지 않는 내용을 말할 가능성이 커집니다.

* 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?

  → 백엔드 실행 환경을 세팅하고 API가 정상적으로 동작하도록 연결하는 부분이 어려웠습니다. 특히 가상환경을 활성화하려고 PowerShell 스크립트 명령어를 실행했는데 보안 정책 때문에 차단되어서, 현재 프로세스 범위에서 실행 정책을 임시로 변경해 해결했습니다.

## 데이터 관련

* 왜 이 데이터를 선택했는가?

  → CareerFit AI는 실제 채용 공고를 기반으로 사용자에게 맞는 공고와 역량을 추천하는 서비스이기 때문에, 기업명, 직무명, 고용 형태, 요구 기술, 자격 요건이 포함된 채용 공고 데이터가 가장 적합하다고 판단했습니다.

* 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?

  → 채용 공고마다 작성 방식이 다르고, 요구 기술이나 자격 요건이 명확하지 않은 경우가 있을 수 있습니다. 또한 오래된 공고나 중복 공고, 일부 정보가 비어 있는 공고는 분석 결과의 신뢰도를 낮출 수 있습니다.

* SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?

  → 두 저장소의 역할이 다르기 때문입니다. SQLite는 기업명, 직무명, 고용 형태처럼 구조화된 데이터를 저장하고 정확하게 조회하기 위해 사용하고, ChromaDB는 공고 내용을 벡터화해 사용자 입력과 의미적으로 비슷한 공고를 찾기 위해 사용합니다.

* 결측치와 중복을 어떻게 처리했는가?

  → 결측치는 먼저 어떤 컬럼에 비어 있는 값이 있는지 확인한 뒤, 분석에 필요한 핵심 정보가 부족한 경우 제거하거나 적절한 기본값으로 처리했습니다. 중복 데이터는 기업명과 직무명 조합을 기준으로 같은 공고인지 확인하고 제거했습니다.

## RAG 관련

* RAG 구조에서 ChromaDB는 어떤 역할을 하는가?

  → ChromaDB는 채용 공고 텍스트를 벡터 형태로 저장하고, 사용자의 입력과 의미적으로 가까운 공고를 검색하는 역할을 합니다. 즉, AI가 답변을 만들 때 참고할 근거 데이터를 찾아주는 검색 저장소입니다.

* n_results 값을 높이면 어떤 트레이드오프가 생기는가?

  → 더 많은 공고를 검색할 수 있어 참고 정보가 늘어나지만, 관련성이 낮은 공고까지 포함될 수 있습니다. 또한 처리해야 할 데이터가 많아져 응답 속도가 느려지고 답변이 산만해질 수 있습니다.

* 검색 결과 품질을 어떻게 검증했는가?

  → 사용자의 전공, 기술, 관심 직무를 입력했을 때 실제로 관련 있는 채용 공고가 검색되는지 확인했습니다. 예를 들어 입력한 기술 스택이나 직무 키워드와 검색된 공고의 내용이 의미적으로 잘 맞는지 비교했습니다.

* Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?

  → Hallucination은 AI가 실제 근거가 없는 내용을 사실처럼 말하는 현상입니다. RAG는 AI가 외부 데이터를 검색해서 그 내용을 바탕으로 답변하게 하므로, 실제 공고에 기반한 답변을 생성해 Hallucination 가능성을 줄일 수 있습니다.

## API 관련

* /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?

  → 사용자의 전공, 보유 스킬, 관심 직무 같은 입력 데이터를 서버로 보내 분석해야 하기 때문입니다. POST는 데이터를 요청 본문에 담아 보낼 수 있어 사용자 입력을 기반으로 분석하는 기능에 적합합니다.

* 응답 JSON에 sources를 포함시킨 이유는 무엇인가?

  → AI가 어떤 채용 공고를 참고해서 답변했는지 보여주기 위해서입니다. sources를 포함하면 사용자가 추천 결과의 근거를 확인할 수 있어 서비스의 신뢰도를 높일 수 있습니다.

* /health 엔드포인트가 왜 필요한가?

  → 서버가 정상적으로 실행 중인지 간단히 확인하기 위해 필요합니다. 배포 환경이나 프론트엔드 연결 과정에서 백엔드 서버 상태를 빠르게 점검할 수 있습니다.

## 보안 관련

* .env 파일을 GitHub에 올리면 어떤 일이 생기는가?

  → API Key 같은 민감한 정보가 외부에 노출될 수 있습니다. 이 경우 다른 사람이 내 API Key를 무단으로 사용하거나 비용이 발생할 수 있기 때문에 .env 파일은 GitHub에 올리면 안 됩니다.

* .env와 .env.example의 차이는 무엇인가?

  → .env는 실제 API Key나 환경변수 값이 들어 있는 개인 설정 파일입니다. 반면 .env.example은 어떤 환경변수가 필요한지만 알려주는 예시 파일이며, 실제 비밀값은 포함하지 않습니다.

## Docker 관련

* Docker를 사용하는 이유는 무엇인가?

  → 개발 환경이 달라도 같은 방식으로 프로젝트를 실행하기 위해 사용합니다. Python 버전, 패키지, 실행 명령어 등을 컨테이너 안에 고정해두면 로컬과 배포 환경에서 발생하는 환경 차이 문제를 줄일 수 있습니다.

* Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

  → EXPOSE는 컨테이너가 사용할 포트 번호를 알려주는 역할을 합니다. CMD는 컨테이너가 실행될 때 어떤 명령어로 서버를 시작할지 지정하는 역할을 합니다.
